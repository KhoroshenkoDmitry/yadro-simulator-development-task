# YADRO - симулятор энергосети

Решение тестовой задачи. Дано:
1. Список потребителей. Про каждого известно почасовое потребление энергии в течение суток.
2. Список генераторов энергии. Про каждый известно: почасовая генерация энергии в течении суток, и стоимость генерации. Есть два вида генераторов: один генерирует одинаковое кол-во энергии в любой час, стоимость высокая (дизель-генератор). Второй генерирует разное кол-во энергии в разные часы, в некоторые часы не работает вообще. Генерация дешевая (солнечные панели).
Надо:
Составить расписание работы генераторов (какие генераторы в какие часы включать), так, чтобы

1. Все потребители в каждый час получали требуемую энергию
2. Общая стоимость генерации была минимальна.
3. Если в какие-то часы всей генерации недостаточно для всех потребителей, то максимизировать количество включенных потребителей на этот час.

Формализацию и решение смотреть в `solution.md`. 
## Структура репозитория

```
src/
    models.py        - Consumer, DieselGenerator, SolarGenerator, ProblemInstance, Schedule
    scheduler.py     - двухфазный MILP-решатель (бэкенды scipy / pulp, стратегии per_hour / monolithic)
    io_utils.py      - загрузка задачи и сериализация расписания в JSON
main.py              - CLI
tests/
    cases/           - генератор тестовых JSON-сценариев и сами JSON
    unit/            - unit-тесты (pytest + coverage)
```

### Почему в репозитории и `pyproject.toml`, и `requirements*.txt`, и `pytest.ini`
 
`uv` удобен, но он не гарантированно установлен у ревьюера/CI. Поэтому в репозитории осознанно дублируется минимум конфигурации, чтобы проект собирался и проверялся **без `uv`**, на голом Python и pip:
 
- **`requirements.txt` / `requirements-dev.txt`** - список зависимостей в формате, который понимает `pip install -r`. Без них пришлось бы ставить `uv`/`hatch`/etc. С ними достаточно `python -m venv + pip install -r`, что есть везде.
- **`pytest.ini`** - конфигурация pytest в формате INI. `pytest` корректно читает её даже на старых версиях, не глядя в `pyproject.toml`. Если бы конфиг лежал только в `[tool.pytest.ini_options]` секции `pyproject.toml`, то на pytest < 6 конфиг бы игнорировался.
- **`pyproject.toml`** - основной источник истины (зависимости, версия Python, конфиг ruff/mypy/taskipy). `requirements*.txt` синхронизируется с ним вручную.
## Запуск
 
### Вариант 1: с `uv`
 
```bash
uv sync --extra dev          # установить рантайм + dev-зависимости
uv run python tests/cases/generate_cases.py    # сгенерировать тестовые JSON
uv run python main.py tests/cases/surplus.json
uv run python main.py tests/cases/deficit.json
```
 
### Вариант 2: без `uv`
 
```bash
python3.12 -m venv .venv
source .venv/bin/activate              # для Windows: .venv\Scripts\activate
pip install -r requirements.txt        # только рантайм
# или
pip install -r requirements-dev.txt    # рантайм + dev (для тестов и линта)
 
python tests/cases/generate_cases.py
python main.py tests/cases/surplus.json
python main.py tests/cases/deficit.json
```
 
### CLI-флаги
 
```bash
python main.py <input.json> [--backend {scipy,pulp}]
                            [--strategy {per_hour,monolithic}]
                            [--output out.json]
                            [--quiet]
```
 
- `--backend` - какой MILP-решатель использовать. По умолчанию `scipy` (HiGHS).
- `--strategy` - `per_hour` решает 24 независимые подзадачи (соответствует постановке), `monolithic` решает всё одним MILP (полезно для будущих расширений с межчасовыми связями).
- `--output` - путь для записи расписания в JSON.
- `--quiet` - не печатать почасовую таблицу.

## Тестовые сценарии
 
Два сценария согласно заданию (10 потребителей, 3-5 генераторов):
 
- **`tests/cases/surplus.json`** - 10 домов, 4 генератора (2 солнечных + 2 дизельных). Суммарная мощность с запасом покрывает пиковую нагрузку. Ожидаемый исход: 100% обслуживания, дешёвые часы покрываются солнцем, ночь - дешёвым дизелем.
- **`tests/cases/deficit.json`** - 10 домов с повышенным потреблением, 3 генератора (2 солнечных + 1 маленький дизель). Ночью и в вечерний пик мощности не хватает. Ожидаемый исход: ~79% обслуживания, чёткие "потерянные" часы у части потребителей.
JSON-кейсы регенерируются скриптом `tests/cases/generate_cases.py` - параметры (число домов, базовое потребление, пики солнца) явно заданы константами в скрипте.


## Допущения и упрощения модели

Перечислены явные упрощения, которые сделаны в модели - каждое можно снять, расширив постановку.

1. **Идеальная надёжность генераторов.** Считается, что включённый генератор всегда выдаёт ровно `E_k(t)`. Нет отказов, нет планового обслуживания, нет деградации со временем.
   *Расширение:* добавить вероятность отказа `p_fail_k` и решать робастную задачу, либо запускать симуляцию Монте-Карло поверх детерминированного решения.

2. **Выдача - детерминированная константа, не интервал.** Реальное солнце даёт `[low, high]` в зависимости от облачности, дизель имеет минимальный устойчивый уровень и переходные режимы.
   *Расширение:* перейти от бинарных `Q_k` к непрерывным `q_k ∈ [Q_min, Q_max]` с дополнительной бинарной переменной "включён/выключен". Стоимость становится `C_fixed_k · on_k + C_variable_k · q_k`.

3. **Нет приоритета потребителей.** Все потребители равноценны, цель - максимизировать их *число*. На практике критическая инфраструктура (больницы, лифты) должна обслуживаться приоритетно.
   *Расширение:* добавить веса `w_j` и максимизировать `sum_j w_j · S_j`. Для жёсткого приоритета - добавить ограничения вида `S_critical = 1` (обслужить безусловно или вернуть невозможность).

4. **Потребитель обслуживается целиком или никак.** Бинарность `S_j` - это упрощение. В реальности возможно частичное обслуживание.
   *Расширение:* заменить `S_j` на непрерывную `s_j ∈ [0, 1]` и интерпретировать как долю обслуженной нагрузки.

5. **Отсутствие накопителей энергии.** Часы независимы - избыток солнечной генерации в полдень не переносится на вечерний пик.
   *Расширение:* ввести переменные `B_t` (заряд батареи в начале часа `t`), связи `B_{t+1} = B_t + charge_t − discharge_t`, ограничения ёмкости и КПД. Это сразу делает задачу неразложимой по часам - пригодится стратегия `monolithic`.

6. **Нет затрат на включение и переходные процессы.** Дизель сейчас можно включать-выключать каждый час бесплатно. В реальности есть стартовая стоимость, минимальное время работы, ramp-up rate.
   *Расширение:* добавить переменные `start_k_t = max(0, on_k_t − on_k_{t-1})`, штрафовать `sum_t start_k_t · C_start_k` в целевой функции. Минимальное время работы выражается дополнительными неравенствами.

7. **Стоимость зависит только от факта включения, не от объёма выдачи.** Сейчас `cost_k(t)` - это стоимость "включения на час", а не функция от `q_k`.
   *Расширение:* кусочно-линейная функция стоимости от объёма, нелинейная (квадратичная) для точной модели расхода топлива → QP/MIQP.

8. **Сутки изолированы.** Решение на день не учитывает ни прогноз погоды на завтра, ни плановое потребление.
   *Расширение:* увеличить горизонт планирования до недели/месяца.

9. **Игнорируется сетевая топология.** Все потребители и генераторы - в одной точке, потерь на передачу нет.
   *Расширение:* добавить узлы и линии с пропускной способностью и потерями.

## Использование ИИ при решении

ИИ (Claude Sonnet/Opus) применялся на следующих этапах:

- **Перепроверка корректности формализации задачи.** Я закидывал в Claude свою формализацию, что отличалась от исходной лишь тем, что вместо верных индексных множеств там были исключительно плейсхолдеры $I$ и $J$. В процессе формализации я несколько раз начинал сначала, из-за чего сбился с подобных деталей - и модель их довольно быстро восстановила. Но толкьо их.
- **Генерация шаблонного кода.** Парсеры JSON, dataclass-модели, скелеты CLI, валидация входа, обёртки над scipy/pulp - это типовой код, ИИ пишет его быстрее, чем я набирал бы вручную.
- **Генерация тестовых случаев.** Сценарии `surplus`/`deficit` с реалистичными профилями (морнинг/ивнинг пики, колокол солнца) сгенерированы ИИ по описанию "10 домов, бытовое потребление, выдержи нужное соотношение мощностей".
- **Рефакторинг по требованиям ruff/mypy.** Прогон ошибок линтера через ИИ с просьбой исправить - заметно быстрее, чем ловить их по одной.

**Инфраструктуру писал сам:** `pyproject.toml`, `pytest.ini`, `requirements*.txt`, GitHub Actions workflow, pre-commit конфиг, разбивка на модули и пакет.

### Примеры промптов

- *"Напиши абстрактный класс `Generator` и два наследника: `DieselGenerator` (константная выдача и стоимость, скаляры в конструкторе) и `SolarGenerator` (почасовые профили длины 24). Сохрани интерфейс `output[hour]`/`cost[hour]` через `@property`, чтобы не ломать существующий solver."*
- *"Вот ошибки mypy в `--strict` режиме: \[список\]. Поправь, не устанавливая дополнительные стабы - используй type: ignore точечно."*
- *"Сгенерируй два JSON-сценария: 10 потребителей с реалистичными профилями. В первом сценарии генерации с избытком, во втором ночью и в пик не хватает."*

### Как проверял сгенерированное

- **Тесты.** Покрытие 96%, 27 unit-тестов. В числе тестов - кросс-валидация: один и тот же сценарий решается двумя независимыми бэкендами (scipy/HiGHS и PuLP/CBC) и двумя стратегиями (per_hour и monolithic) - расхождение по числу обслуженных или по итоговой стоимости считается ошибкой. Это ловит ошибки формализации, потому что вероятность одинаково ошибиться в обоих бэкендах низкая.
- **Феасибилити-проверка.** В каждом тесте каждое решение проходит явную проверку энергобаланса по часам.
- **Чтение каждой строки.** Несмотря на удобство, всё же ИИ до сих пор пишет код очень плохо. Зачастую есть стилистические ошибки а-ля первокурсник вроде однобуквенного нейминга - и это самое приятное из того, что может быть и что может быть объяснено просто малым размером контекста. Код, конечно, некрасивый, но хотя бы рабочий и проходит тесты. Но бывали случаи, когда после написания кода моделью всё просто переставало работать. Например, было развлечение переименовывать модули на своё усмотрение без лишних комментариев - потом ищи, кто куда подключался и как кого правильно назвать. 
- **Линт + типы + форматер.** `ruff check`, `mypy --strict`, `ruff format` гонялись как контроль качества. ИИ-сгенерированный код проходил синтаксис, но не проходил строгий линт.

### Где ИИ был особенно полезен

Типовая разметка (форматы JSON, заголовки тестов, parametrize-таблицы), скелеты dataclass'ов с валидацией в `__post_init__`, рутинная подгонка под линтер.

### Где ИИ был очень бесполезен (ну очень)

Всё, что касается архитектурных решений, аккуратного нейминга, в целом красоты кода - это сплошная головная боль, перепроверка и переписывание за моделью. Я уже приводил пример с неймингом, но вот пример и с архитектурой: Claude иногда игнорировал архитектуру классов, заданных мной, и превращал абстрактный `Generator` и его наследников в один общий класс `Generator` просто с полем `kind: GeneratorKind`, где `GeneratorKind` - это `enum`. Конечно, такое решение накладывало огромные ограничения и обязанности для пользовательского кода: от необходимости хранить огромное количество лишней информации для дизель-генератора и до ограничений функциональности новых кастомных генераторов. 

В итоге ощущение такое: ИИ ускоряет на 30–40% на типовом коде, но **не** заменяет ревью. Время на чтение и проверку сгенерированного - реально, и его нужно закладывать в оценку.

## CI

GitHub Actions запускает на каждый push в `main`/`develop` и на каждый PR:

1. `ruff format --check` - проверка форматирования.
2. `task lint` - `ruff check`.
3. `task typecheck` - `mypy --strict`.
4. `task test` - `pytest` с покрытием.

Workflow в `.github/workflows/ci.yml`.
